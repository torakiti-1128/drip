# 要件定義書

## 1. 機能要件 (Functional Requirements)

### 1.1 Intake & Manage（日報入力・管理系）
フォーム入力から構造化データを生成し、Markdownとして外部出力する。

| 機能名           | 詳細要件                                                                                                                                       | コンサルタントの視点（実装の急所）                            |
| :------------ | :----------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------- |
| **日報入力フォーム**  | ・**基本情報**: 日付、作成者、開始/終了時刻、休憩時間（稼働時間を自動計算）。<br>・**業務内容**: タグ、要約(1行)、予定時間、実績時間、ステータス(完了/順調/遅れ)、完了予定日時。<br>・**所感/メモ**: タイトル+本文の複数セット（動的追加可能）。 | UIはSPAで構築し、入力中の状態喪失を防ぐ（LocalStorage等への自動退避）。 |
| **マスタデータ管理**  | 業務・分析タグ（例：要件定義, 実装, テスト, 障害対応, 自己研鑽, マネジメント）の追加・編集。                                                                                        | タグはMECE（モレなくダブりなく）を意識した初期セットを用意する。           |
| **リストビュー**    | 日報一覧表示。日付、稼働時間、サマリー、ステータス（一時保存/完了）の表示。<br>タップで詳細・編集画面へ遷移。                                                                                  | 「未読/既読」や「一時保存」のステータス管理をDBのフラグとして厳格に持つ。       |
| **ファイル出力・連携** | フォーム入力内容を指定のMarkdownフォーマットへ変換[[フォーマット](./../06_DAILY_REPORT_DESIGN/README.md)]。<br>自動命名ルール（`YYYY/MM/DD.md`）の適用。<br>DLボタン、G-Drive等指定URLへのリンクコピー。                                   | G-Drive連携は「アプリからG-Driveへの一方通行のPush」に留め。      |

### 1.2 Batch & AI Filter（定期解析系）
蓄積された構造化データを指定期間で一括で読み込み、AIへ渡す。

| 機能名          | 詳細要件                                                                                      | コンサルタントの視点（実装の急所）                                        |
| :----------- | :---------------------------------------------------------------------------------------- | :------------------------------------------------------- |
| **スケジューラ**   | 1週間、1ヶ月など、指定したサイクルで未処理の完了済み日報をバッチ処理。                                                      | cron等による定期実行。処理の冪等性（何度実行しても結果が同じ）を担保する。                  |
| **AI解析エンジン** | 複数日分の日報をコンテキストとしてGemini等へ投入。<br>・全体概要、良かった点、悪かった点<br>・改善アクション（5〜10個）<br>・定着クイズ（5〜10問）を生成。 | 1ヶ月分の日報はトークン数が膨大になる。不要な情報（休憩時間等）は削ぎ落としてAIに渡すプロンプト最適化が必須[[プロンプト例](./../05_PROMPT_DESIGN/README.md)]。 |
| **自動タグ付け**   | 生成されたアクションとクイズに対し、最適なタグ（例：コーディング、コミュニケーション）をAIに分類させる。                                     | AIの出力ブレを防ぐため、事前に定義した「マスタデータ」のタグリストの中から選ばせるプロンプトにする。      |
| **ファイル出力・連携** | フォーム入力内容を指定のMarkdownフォーマットへ変換[[フォーマット](./../07_AI_OUTPUT_DESIGN/README.md)]。<br>自動命名ルール（`YYYY/MM/DD-YYYY/MM/DD.md`）の適用。<br>DLボタン、G-Drive等指定URLへのリンクコピー。                                   | G-Drive連携は「アプリからG-Driveへの一方通行のPush」に留め。      |

### 1.3 Discharge & Action（出力・学習定着系）
生成されたバッチ結果を消化し、行動変容へ繋げる。

| 機能名 | 詳細要件 | コンサルタントの視点（実装の急所） |
| :--- | :--- | :--- |
| **解析結果リスト** | バッチ処理結果の一覧。出力日時、概要、リンク、未読/既読ステータス。<br>Slack/Google Chat等への自動通知。 | 既読フラグが立っていないものを、毎日リマインド通知する「執着」の仕組みを入れる。 |
| **アクション・クイズ受講** | 解析結果のタップで、全体サマリーと改善アクションの確認UIを表示。<br>クイズの受講（1問ずつ表示、正誤判定、解説表示）。 | クイズの回答結果（正解/不正解）をDBに記録し、ダッシュボードでの集計用データとする。 |

### 1.4 Dashboard（ダッシュボード・可視化系）
自己成長の度合いと、システムの利用状況を定量化する。

| 機能名 | 詳細要件 | コンサルタントの視点（実装の急所） |
| :--- | :--- | :--- |
| **KPIトラッキング** | ・**日報作成率**: 営業日（土日祝除外）に対する作成割合。<br>・**クイズ着手率**: 生成されたクイズ数に対する回答済み割合。<br>・**正答率**: タグ・アクションごとのクイズ正答率。<br>・**弱点分析**: 改善アクションに最も多く付与されたタグのランキング。<br>・**文字数推移**: 日報の平均文字数と時系列グラフ。 | 計算ロジックが複雑になるため、フロントエンドで計算するのではなく、Rails側で集計済みのJSONを返すAPIエンドポイントを設計する。 |

### 1.5 Administration（管理・設定系）

| 機能名 | 詳細要件 |
| :--- | :--- |
| **プロンプト＆AI設定** | 解析用プロンプトの編集。利用するLLM（Gemini 2.0 Flash等）の切り替え。 |
| **フォーマット設定** | Markdown出力時の見出し構成や、保存先URLテンプレートの編集。 |
| **通知設定** | Webhook URLの登録、通知タイミング（即時、朝8時等）、通知メッセージフォーマットの変更。 |

## 2. 非機能要件 (Non-Functional Requirements)

| カテゴリ | 要求水準・制約 |
| :--- | :--- |
| **データ完全性** | システムの単一障害点（SPOF）を回避するため、データの「正」はPostgreSQLに置く。Google Drive上のファイルが手動で削除・変更されても、システム本体に影響を与えない設計とする。 |
| **パフォーマンス** | ・日報保存・編集等のUI操作：1秒以内。<br>・バッチ解析（AI処理）：非同期処理とし、完了次第通知（最大待機時間目安：3分以内）。<br>・ダッシュボード表示：複雑な集計クエリとなるためキャッシュを適切に利用し、2秒以内に描画。 |
| **可用性・保守性** | 完全個人運用を前提に、Frontend は Vercel、Backend は Fly.io、DB は Supabase を利用する。エラー発生時（パース失敗、API制限等）はSlack等にアラートを飛ばし、手動でリトライ可能な設計とする。 |
| **UI/UX** | PCでの入力を主軸としつつ、ダッシュボードの確認とクイズの受講はスマートフォン（PWA）から片手でストレスなく行えるレスポンシブデザインとする。 |
